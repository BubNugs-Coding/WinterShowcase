SantaSurvival

GitHub repo

What it is

A side-scrolling web game where you control Santa to deliver presents while dodging missiles. Built on Phaser 3 and served as a static site (dev uses a simple HTTP server).

  • Core loop: deliver presents → score climbs → survive threats.
  • Input: mouse movement controls Santa’s vertical position.
  • Leaderboard: optional Firebase name-only score submission.
How to run locally
  1. Install dependencies: npm install
  2. Start server: npm start (serves on http://localhost:8080)
  3. Open the URL in your browser.

The game loads Phaser via CDN and runs src/main.js as an ES module.

Leaderboard setup (optional)

The project supports a lightweight Firestore leaderboard (no accounts). It’s meant for fun, not anti-cheat.

  1. Create a Firebase project + Web App.
  2. Enable Firestore.
  3. Paste Firestore rules from the repo’s setup doc.
  4. Add your Web App config to src/services/firebaseConfig.js.

Tech & structure

Engine
Phaser 3
Frontend
Vanilla JS modules
Server
Static HTTP server (dev: http-server)
Persistence
Firebase Firestore (optional)
Controls & gameplay notes
  • Move: move mouse up/down to control Santa’s Y position.
  • Goal: fly over houses to deliver presents and score points.
  • Threat: avoid missiles (3 lives).

PixInsightScripts-BubNugs

GitHub repo

What it is

A PixInsight PJSR script that acts like a guided wizard for processing linear LRGB master frames. It expects four open images named exactly L, R, G, B.

Install & run (quickstart)
  1. Copy LRGB_Wizard.js into your PixInsight scripts directory.
  2. Refresh scripts (or restart PixInsight).
  3. Open your masters named L, R, G, B.
  4. Run: Script → LRGB Processing Wizard.
Workflow steps (expand for the full list)
  • Pre-flight: checks required images and sets reference.
  • DynamicCrop: manual crop on L, then apply to R/G/B.
  • AutoDBE: automatic gradient removal.
  • PCC: photometric color calibration.
  • STF → HT: convert auto-stretch into histogram transform.
  • LRGBCombination: merge L into RGB with adjustable weights.
  • SCNR: remove green cast.
  • Histogram/Curves: black point + contrast/saturation.
  • LHE / Star reduction / HDRMT: detail + star control.
  • Export: final save + optional JPG/TIFF exports.

Why a wizard?

PixInsight has powerful tools, but a repeatable workflow can be hard to remember. This script makes the pipeline concrete and teachable: one step at a time, with clear expectations and optional skipping.

  • Guided UX: dialogs explain what to do and what “good” looks like.
  • Consistency: applies the same crop/choices across channels.
  • Practical automation: reduces the “blank page” feeling in processing.
How it works (high-level)
  • Pre-flight scan: walks open PixInsight windows and matches IDs.
  • Stateful pipeline: stores view/image refs for later steps.
  • Non-modal steps: lets you interact with the image while dialogs are open.

3DWizard

GitHub repo

What it is

A static, client-side web app for 3D printing/model utilities. The goal is to provide pre-slicer inspection and helpers—everything runs in the browser (no backend).

  • In-browser processing: no uploads, no server-side compute.
  • Toolbox mindset: focused utilities instead of a single feature.
  • Modern stack: Vite + React + TypeScript.
Highlighted tool ideas
  • Mesh viewer/inspector: preview, stats, wireframe toggles.
  • Sanity checks: non-manifold edges, flipped normals, open boundaries.
  • Repair/cleanup: common “one click” fixes and optimizations.
  • Conversions: STL/OBJ/GLTF (limitations noted in the repo).

How to run

Development
  1. npm install
  2. npm run dev
Build for production
  1. npm run build
  2. Deploy the generated dist output.

For GitHub Pages deployments, the repo’s docs mention setting the Vite base path to the repo name if needed.

Pomfret Observatory / N.I.N.A. Automation Milestone

A short timeline of a real integration effort: I started the N.I.N.A. connection work, hit major blockers, someone else took over for a while, and now the system is running and I’m rejoining for the next steps toward automated remote operation.

Timeline

  1. Kickoff: initial N.I.N.A. ↔ observatory connection effort
    I began the process of connecting N.I.N.A. to the Pomfret Observatory hardware/software stack.
  2. Blockers: real-world integration issues
    I ran into multiple issues typical of observatory integration work (port/driver conflicts, mismatched site settings, and intermittent device connectivity).
  3. Handoff period
    Another person took over the project for a while and pushed it forward.
  4. Now: running + ready for the next phase
    The system is now operating, and I’m joining back in to drive the next steps: automation and remote operation.
Remote automation & N.I.N.A. sequencers

Working on implementing remote automation using N.I.N.A.'s powerful sequencing system:

  • Basic Sequencer: Simple sequential instruction lists for straightforward imaging runs. Good for single-target sessions with standard procedures.
  • Advanced Sequencer: Complex conditional logic, triggers, and parallel operations. Enables smart automation with conditions (weather checks, altitude limits, time-based actions), loop structures, and error recovery paths for unattended remote operation.
  • Remote operation goals: Reliable startup/shutdown procedures, automated target selection, safety monitoring, and recovery from common failures—all essential for running the observatory remotely.
What "next steps" means (automation)
  • Reliability: keep connections stable over long sessions.
  • Procedures: startup/shutdown checklists translated into automation.
  • Safety mindset: remote operation should fail safe and be recoverable.
  • Scheduling: repeatable imaging plans and unattended runs.
Concrete “PDF snapshot” highlights (what I was dealing with)
  • Connection failures: “Could not open communications port… Error = 201.”
  • Settings drift: N.I.N.A. prompting to synchronize site settings (lat/long/elevation mismatch).
  • Mixed stability: “Telescope connected”, “Dome connected”, then “Focuser connection lost”.
  • Real integration complexity: multiple devices (mount, camera, dome, focuser, filter wheel) and drivers.

The point of including this: even “simple” automation depends on reliable low-level connectivity and consistent configuration.

What this demonstrates

  • Debugging: diagnosing failures across software + hardware boundaries.
  • Systems integration: coordinating drivers, settings, and real devices.
  • Collaboration: a handoff happened, the project progressed, and I’m returning to continue it.
  • Documentation: turning experience into a clear narrative and plan.
System elements (high-level)
  • Orchestrator: N.I.N.A. (imaging session control)
  • Mount control: TheSky / driver layer (mount not always recognized when ports are busy)
  • Devices: mount, dome, focuser, camera/filter wheel (each can fail independently)

This is intentionally high-level (no sensitive remote-access details), focused on what I learned from the integration.
